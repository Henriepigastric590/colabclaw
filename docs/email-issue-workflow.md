# Email → Issue → Draft Workflow

이 문서는 Gmail 메일을 GitHub Issue로 전환하고, 답장 초안을 자동 생성한 뒤 사람 승인 후 발송하는 운영 방식을 정리한다.

## 목표

- 메일 1개당 이슈 1개 생성
- 메일 내용 요약 자동 생성
- 답장 초안 자동 생성
- 실제 발송은 사람 승인 후에만 수행
- 이슈/댓글/라벨을 활용해 메일 상태를 추적

## 기본 원칙

### 자동 수행
- 메일 읽기
- 메일 분류
- 이슈 생성
- 답장 초안 생성

### 수동 승인 필요
- 초안 확정
- 외부 메일 발송

기본 정책은 **"자동은 초안까지, 발송은 승인 후"** 이다.

## 데이터 구조

### 메일 1개 = 이슈 1개

이슈 제목 예시:
- `[메일] AI 프로바이더 관련 문의`
- `[메일] Colab 구독 취소 안내`

이슈 본문에 포함할 정보:
- 원본 메일 제목
- 보낸사람
- 수신 시각
- 요약
- 중요도
- 카테고리
- 답장 필요 여부
- 현재 상태

예시:

```md
## 메일 정보
- 원본 제목: [AIF]AI 프로바이더 관련 문
- 보낸사람: 류태욱 <tu.ryu@aifactory.page>
- 수신 시각: 2026-03-25 17:08 KST

## 요약
Amazon Bedrock API 호출 시 Too Many Tokens 오류가 발생했고,
첨부 이미지와 함께 확인 요청이 들어옴.

## 분류
- category: work
- priority: high
- reply-needed: yes
- status: draft-pending
```

## 댓글(Comment) 사용 방식

### AI 댓글
이슈 생성 직후 AI가 댓글로 초안을 남긴다.

예시:

```md
## Draft Reply v1

안녕하세요, 류태욱님.

전달주신 내용 확인했습니다.
Amazon Bedrock API 사용 중 Too Many Tokens 오류가 발생한 상황으로 이해했습니다.

먼저 첨부 자료를 확인해 원인을 검토하겠습니다.
추가로 요청 파라미터 또는 사용한 모델 정보가 있다면 함께 공유 부탁드립니다.

확인 후 다시 안내드리겠습니다.
감사합니다.
```

### 사람 댓글
사용자는 이슈 댓글 또는 채팅으로 아래와 같이 수정 요청 가능:
- `더 부드럽게 수정`
- `일정 안내 한 문장 추가`
- `사과 표현은 빼고 기술적으로만`
- `이 버전으로 확정`

AI는 새 댓글로 `Draft Reply v2`, `v3` 형태로 응답한다.

## 라벨 설계

### 기본 라벨
- `email` : 이메일 기반 이슈
- `needs-reply` : 답장 필요
- `draft-ready` : 초안 생성 완료
- `waiting-human` : 사람 검토 대기
- `approved` : 발송 승인 완료
- `sent` : 발송 완료

### 분류 라벨
- `work`
- `billing`
- `promo`
- `sales`
- `bug`
- `support`

### 우선순위 라벨
- `priority:high`
- `priority:medium`
- `priority:low`

## 상태 흐름

1. 새 메일 수신
2. 이슈 생성 (`email`, `needs-reply`)
3. AI 요약 + 초안 댓글 생성 (`draft-ready`, `waiting-human`)
4. 사람 피드백 반영
5. 사람 승인 (`approved`)
6. 실제 발송
7. 발송 후 `sent` 라벨 및 댓글 기록

## 승인 규칙

기본적으로 아래 중 하나가 있을 때만 발송 가능:
- 채팅에서 `그대로 보내`
- 채팅에서 `수정해서 보내`
- 이슈 댓글에서 `승인`, `SEND`, `approved`
- 특정 승인 라벨 추가

## 안전 규칙

- 광고/뉴스레터는 기본적으로 초안 생성 생략 가능
- 민감한 메일은 자동 발송 금지
- API 키, 토큰, 개인식별정보는 이슈에 직접 기록하지 않음
- 첨부파일 원문은 가능한 한 요약만 기록
- 실제 발송 전에는 항상 최종 본문/수신자/thread 확인

## 추천 초기 운영안

### 1단계
- 메일 → 이슈 생성
- 요약 작성
- 초안 댓글 작성

### 2단계
- 사람 수정 요청 반영
- 댓글 버전 관리

### 3단계
- 승인 시만 발송
- 발송 로그 댓글 남기기

## GitHub 활용 포인트

- Issues: 메일 단위 작업 추적
- Comments: 답장 초안/수정 이력 저장
- Labels: 상태/우선순위/카테고리 관리
- Projects: Inbox → Draft Ready → Waiting → Approved → Sent 흐름 구성

## 예시 운영 문장

- `이슈 생성 후 Draft Reply v1 자동 코멘트`
- `사람 피드백 수집`
- `최종 승인 전에는 절대 발송하지 않음`
- `발송 후 sent 라벨 부여 및 타임스탬프 기록`
